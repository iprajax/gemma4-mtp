01Why this matters for on-device AI

The case for running language models locally instead of through an API is no longer hypothetical. Privacy: your prompts never leave the device. Cost: after the model download, inference is free. Latency: no round-trip to a datacenter. Reliability: works on a plane, in a metro, in a Bangalore power cut. The block on adoption was always speed — local LLMs have historically been 5–10× slower per token than hosted equivalents.

Speculative decoding is the technique that closes that gap most cleanly. A small "drafter" model proposes k tokens; the big "target" model verifies them in a single parallel forward pass; rejected tokens are discarded. At temperature=0 the output is provably identical to running the target alone. You're paying nothing in quality and getting up to 2× more tokens per second for free.

This is exactly the regime where on-device wins: batch=1, single user, no API quota. Which is why I cared about Google's announcement enough to spend a Saturday measuring it.

02What Google announced, and what I wanted to verify

"Multi-Token Prediction (MTP) drafters... deliver up to a 3× speedup without any degradation in output quality."

"the 26B mixture-of-experts model presents unique routing challenges at a batch size of 1 on Apple Silicon... batch sizes of 4 to 8 unlock up to a ~2.2× speedup locally."

That second quote is the interesting one. It implies that on Apple Silicon at batch=1 — which is every single-user laptop workload — the speedup is muted. I wanted to know exactly how muted, and whether the runtime choice changes the answer. It does. Dramatically.

Three runtimes can run Gemma 4 locally on a Mac today: Hugging Face transformers with MPS, Google's own litert-lm, and llama.cpp. I tested all three. Here is the same machine, the same prompts, temperature=0, real outputs (not synthetic random-token decode):

03My Mac, exactly

Every benchmark in this post was measured on a single machine. Not a Mac Studio, not an M4 Max — the mid-tier MacBook Pro that most working developers actually carry:

One detail matters for context: Google's published benchmark on Gemma 4 E2B GPU shows 160 tok/s. That measurement was on an M4 Max, which has roughly 1.7× more GPU cores than my M4 Pro. My E2B baseline of ~84 tok/s is not a regression against Google's 160 — it's the same runtime on smaller silicon. Speedup ratios should hold; absolute throughput won't.

04How I benchmarked

I wrote two scripts. Both are in the repo:

  • bench_mtp.py — Hugging Face transformers with MPS. Loads target + drafter once, sweeps k ∈ {2, 4, 6, 8}, runs paired baseline / +MTP on each prompt, hash-checks output equality at greedy.
  • bench_litertlm.py — subprocess wrapper around litert-lm run. The bundled litert-lm benchmark is unsuitable for measuring MTP (see Gotcha 2). My script runs real prompts via run, times wall-clock, and counts output tokens with the matching Hugging Face tokenizer to compute tok/s.

For each (runtime, model) pair I primed the program cache with two warmup invocations per mode, then measured three real prompts: a Python coding task (~700 tokens out), a step-by-step word problem (~1000 tokens out), and a short IDE-style completion (~80–150 tokens out). All at temperature=0. Numbers in the table are arithmetic means over those three prompts.

05Two gotchas that almost cost me the right answer

Gotcha 1 · Stale bundle silently disables MTP

Google re-cut the litert-community/gemma-4-*-litert-lm bundles on May 5 to add MTP wiring metadata. Bundles downloaded earlier load the drafter weights but the engine config silently stays enable_speculative_decoding: false — even when the CLI flag passes true. My first run was on a stale bundle and I was about 30 minutes into writing "MTP is a no-op on Mac" before I noticed. Re-pull the bundle.

Gotcha 2 · The bundled benchmark uses random tokens

litert-lm benchmark decodes random tokens for its synthetic decode loop. Drafter acceptance rate on random sequences is essentially 0%, so MTP becomes pure verification overhead and shows as a regression. A correctly engaged MTP path reads as 0.93× through this tool. Use real prompts via litert-lm run, externally timed. That's what bench_litertlm.py does.

06Reproduce it in 5 minutes

Install:

# Python 3.12+ recommended
git clone https://github.com/iprajax/gemma4-mtp
cd gemma4-mtp
uv tool install litert-lm

Run the LiteRT-LM benchmark on E2B (smaller, faster first run):

python bench_litertlm.py --model e2b

Or run a single direct invocation to see the engine flag flip:

litert-lm run \
  --from-huggingface-repo=litert-community/gemma-4-E2B-it-litert-lm \
  --backend=gpu \
  --enable-speculative-decoding=true \
  --temperature=0 \
  --verbose \
  --prompt="Write a Python function that returns the longest strictly increasing subarray." \
  2>&1 | grep "Speculative decoding"
# Must print: Speculative decoding : true
# If it prints "false", your bundle is stale — re-pull.

07What to do, what to skip on a Mac

Do
  • Use LiteRT-LM v0.11.0 Metal GPU + MTP for the fastest local Gemma 4
  • Re-download bundles dated before 2026-05-05
  • Pick E4B over E2B if your machine has the RAM — bigger speedup
  • Run real prompts to measure speed; --verbose to confirm
Don't
  • Trust litert-lm benchmark for any MTP measurement
  • Expect the headline 2× on transformers+MPS — you'll get 1.04–1.24×
  • Wait for llama.cpp + MTP — gemma4_assistant isn't merged yet
  • Use MTP on E2B in transformers — on a small target it can be a net loss

08Why one runtime wins

The technique is identical across runtimes — same drafter weights, same verify step. The runtime carries the rest. LiteRT-LM beats transformers on identical hardware for three reasons:

  • int4 weights instead of bf16. Each target step is faster, leaving more room for accepted-draft savings to register.
  • Purpose-built Metal kernel chain. WebGPU sampler + drafter pipeline are designed for this exact pattern.
  • No Python in the hot loop. transformers' assistant_model wrapper pays per-step Python and bf16-on-MPS overhead. LiteRT-LM stays in compiled C++.

This is the most useful single takeaway from the experiment. For on-device LLM inference, runtime choice is at least as important as model choice. Two runtimes, same hardware, same model, same technique — and one delivers Google's headline while the other delivers a fifth of it.

Bottom line for on-edge LLMs

Google's MTP announcement is real and measurable on Apple Silicon — at batch=1, on a 24 GB laptop, today. The only conditions are using the runtime they shipped for it, with the bundle they re-cut for it. ~100 tok/s on Gemma 4 E4B. That's the on-device language model speed crossing into "feels like a hosted API" territory, on a single MacBook, with no network. The post most people will write next week says MTP doesn't work on Mac. The data says it works better than any other path on Mac.
